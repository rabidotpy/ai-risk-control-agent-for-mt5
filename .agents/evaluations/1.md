# Strict full evaluation, in plain English

I will go layer by layer. For each one I explain what is good, what is broken, and why it matters. I will keep words simple and explain anything technical the first time I use it.

## 1. The PRD (the requirements document)

The PRD is the document K wrote that says what the system should do. It reads more like a wish list than a clear plan. That causes problems.

**It contradicts what we actually built.** The PRD says use FastAPI, Postgres, and OpenAI. We use Flask, MongoDB, and Claude. None of our choices are wrong, but the PRD was never updated. So if you read it today you will think we are using the wrong stack. We should add a note at the top of the PRD that says "real choices live in context.md".

**The risk rules are vague.** One rule says "profitable trades within quote spike window >= 60%". But the PRD never says what a "quote spike window" is. Another rule says "positive slippage 3 times higher than the peer average". A peer average means comparing this account to other similar accounts (same group, same balance range, same trading style). The PRD does not say what makes an account a peer. So we had to drop those rules and use simpler ones. That is a reasonable choice but K should sign off.

**The PRD lists 5 trigger types but we only use 1.** Trigger types are the reasons that start a scan. The PRD lists scheduled scan, withdrawal request, abnormal profit, high frequency, and news window. Today only "scheduled scan" works. The other four are placeholders. The PRD should say which ones are MVP and which ones come later.

**The PRD writes thresholds for 24 hours but our pulls are 6 hours.** A rule like "100 trades in 24 hours" cannot be checked from a 6 hour pull. We worked around this by scaling down to "25 trades in 6 hours". But trading is not even across the day. A trader does most of their trades during their local market hours, not spread across all 24 hours. So 25 trades in any 6 hours is not the same as 100 in a full day. The honest fix is to add up our 6 hour pulls in our own database to get a real 24 hour view. That is planned for Phase B.

**The PRD has SQL table designs but we use MongoDB.** MongoDB does not use tables. So §11 of the PRD is misleading. Same fix as above: update or delete it.

**The PRD lists 9 helper agents. We built 1.** Email Draft Agent, CS Follow-up Agent, Audit Agent, and others are listed but not built. A reader will think they exist.

**Bottom line.** The PRD is treated like an inspiration document, not a contract. We have started writing context.md as the real spec instead. That works for now but it means the PRD slowly becomes wrong. Either fix the PRD or put a clear sign on it that says "for the real spec, read context.md".

## 2. The overall design

**The current design works for one account at a time but will break under real load.** Today we send one account to the engine, the engine asks Claude four questions in parallel, and Claude answers in a few seconds. Fine. But Phase B has thousands of accounts to scan four times a day. If we run them one after the other at 3 seconds each, 1000 accounts takes 50 minutes. We have no queue, no worker pool, no batch mode. The current shape needs a rethink before the scheduler lands.

**The HTTP endpoint is open to anyone who knows the URL.** There is no API key, no IP allow list, no rate limit. If Alex's backend will call us we need at least an API key. If the endpoint is meant to be private (only inside our network) we should say so clearly.

**There are no logs.** When something goes wrong in production we cannot find out what happened because nothing is recorded. We need at least basic logging. The PRD even mentions Sentry and Prometheus (tools that record and alert on errors) but we have not added either.

**The engine does not protect itself when one Claude call fails.** We ask Claude four questions in parallel (one per risk type). If even one of those four calls fails, the whole analysis for that account dies and we get nothing back. We should catch the error per risk and return a "errored" result for that one risk only. This is the single biggest production bug waiting to happen.

**The "rules in Claude, scoring in Python" split is the best decision in the project.** Claude reads the data and judges each rule (true or false). Python counts the trues and computes the score. This is auditable, repeatable, and the LLM still gives a nice plain English summary. Keep this.

## 3. The data shape (schemas)

**We accept any extra field without complaining. That is dangerous.** Today if Alex sends a typo, like `swap` instead of `swaps`, our code silently treats `swaps` as 0 for every trade. Swap arbitrage detection then stops working. We would never see an error. The fix is to tell Pydantic to reject unknown fields on the inner row models (Trade, Deposit, Withdraw, Bonus). Keep the loose mode only on the outer envelope where Alex might genuinely add new fields.

**One Trade field (`entry`) is required but is just informational.** If Alex ever forgets to send it, validation fails with a 400 error. Either give it a default value of 0 or tell Alex it must always be there.

**The `Trade.id` field is unclear.** MT5 has three different IDs (Deal ID, Order ID, Position ID). Our code uses one of them but the comment is fuzzy. Pick one (Position ID is the right one because we treat each row as one closed position) and write it down.

**We do not check that bonus events and trades fall inside the time window.** If Alex's pull leaks a bonus event from yesterday into today's data, every "in window" rule is silently wrong. A simple filter or a check inside `bucket_by_login` would catch this.

**The evidence dictionary has no type discipline.** Evidence is the dictionary of values we send back, like `{"trade_count_6h": 86}`. The values can be float, int, string, or null. Anything reading this (the Telegram bot, dashboards) has to guess. Worth typing properly later.

**Confirm with Alex which sign is positive on `swaps`.** Our rules treat `swaps > 0` as the trader earning carry. If Alex sends positive numbers when the broker collects (the opposite view), every swap arbitrage rule will work backwards. Worth one line in the message to him.

## 4. The engine and the rules

**The score math is correct.** No issue. Tested well.

**The metric name parser is fragile.** Our rule strings always have spaces around the operator, like `trade_count_6h >= 30`. The parser splits on those spaces. If anyone ever writes a rule without spaces (`trade_count_6h>=30`) the parser silently keeps the whole string as the metric name. Add a normalisation step.

**The parser does not understand `!=`.** No rule uses it today. If you ever do, parsing breaks silently.

**Latency Arbitrage R3 and R4 are not what the PRD says they are.** PRD says one thing, code does another. The replacement rules are reasonable but K has not approved this in writing. Get one line from him before alerts go live, because the difference between "we shipped the PRD" and "we shipped a different thing" matters a lot if a client argues with us.

**Scalping R1 has the same proportional scaling problem as the PRD.** Provisional, not equivalent.

**Bonus Abuse R2 has a math problem.** The rule asks "are there 30 or more trades in the 24 hours after the bonus arrived?" But our data window is only 6 hours. So the rule can only ever see 6 hours of trades after the bonus, not 24. Asking for 30 trades in 6 hours is a much harder bar than 30 in 24 hours. Either lower the threshold to something realistic for 6 hours (around 8) or wait for Phase B when we can actually look at 24 hours of history.

**Swap Arbitrage R2 fires more often in some scan windows than others.** The rule asks "did any position span midnight UTC?" A position closing in the 00:00 to 06:00 window probably opened the day before, so it spans midnight. A position closing in the 12:00 to 18:00 window probably opened the same day, so it does not span midnight. So the rule is biased by the time of day we scan. Not a bug but worth knowing.

**The repeated pattern rule uses raw float comparison.** It groups trades that share `(volume, stop_loss, take_profit)`. If Alex ever sends 0.10 in one trade and 0.10000001 in another (which can happen with float math), they would not group together. In real MT5 data this is unlikely but worth a small tolerance check.

**The "pattern bucket" needs at least 3 identical trades to count.** A trader doing 14 pairs of identical trades (28 trades total, 14 buckets of 2) would not fire the rule because no bucket reaches 3. That is a real false negative we should think about.

**The engine never checks that Claude returned the right number of evaluations.** If Claude returns 3 answers when we asked 4 questions, we score on the 3 we got. We should pick a behaviour. Either treat missing rules as false (safer) or as "insufficient data" and complain (cleaner). Today we silently use what we got.

**There is no test that running the same input twice gives the same score.** With Claude in the loop the answer can drift slightly between runs. We should add a test that mocks Claude with a fixed reply and confirms the engine produces identical output two times in a row.

## 5. How we talk to Claude

**The evaluator code is small and clean.** Good.

**`max_tokens=1024` might be too small for Bonus Abuse.** Bonus Abuse has 5 rules instead of 4, so the response is longer. Probably still fits but we have not tested the worst case. Either bump to 2048 or write a test.

**Prompt caching is set up correctly.** Claude lets us mark big parts of the prompt as cacheable, so repeated calls with the same prompt only pay the full token cost once. We do this. Good.

**No retry policy is set.** The Anthropic SDK retries 2 times on errors by default. For a four times a day scan over thousands of accounts, that is probably not enough during peak Anthropic load. We should set the retry count and wait time explicitly.

**No way to track Claude cost or speed.** When the bill explodes or response time creeps up, we will not know until something breaks.

**The error path is harsh.** If Claude does not return the expected `tool_use` block, we raise an exception. That exception bubbles up and kills the whole account analysis (see the "engine error containment" point above).

## 6. The HTTP API

**`/analyse_risk` does too much in one call.** It validates the input, asks Claude four questions, writes to MongoDB, fires the callback to Alex, and only then answers the original caller. The callback alone can take 10 seconds. If the original caller times out and retries, we run everything twice. The database write is safe to repeat (it just overwrites) but the callback fires twice.

**`/healthz` is a fake.** It always returns "ok" no matter what. It does not check if MongoDB is reachable or if our Anthropic key is valid. A real health check would test both.

**`/analyses` returns 404 if no rows exist.** Fine. But if some risk types are saved and some are not (from a partial failure in a previous run) the endpoint returns whatever it finds with no warning. We should signal "incomplete".

**No auth.** Already mentioned.

## 7. The database

**The unique key is right.** We index on `(mt5_login, start_time, risk_type)` and use upsert (update or insert). Re-runs overwrite cleanly.

**`trigger_type` is not part of the unique key.** That means a manual run will replace a scheduled run for the same window. Probably fine but worth a deliberate decision. If you want both to live for audit, add `trigger_type` to the unique key.

**Nothing ever gets cleaned up.** At 4 risks times 1000 accounts times 4 scans a day that is 16,000 rows a day or about 6 million a year. Local Mongo will be fine. Production should add either a TTL (auto delete after X days) or an archive job.

**`created_at` is misleading.** When we overwrite a row, the `created_at` field is replaced with the current time. So it is actually "last run at", not "first created at". Either rename it or add a separate `updated_at`.

**Two helper functions are dead code.** `has_results()` is only used by tests. `is_test_mode()` is not used anywhere. Either delete them or document why they exist.

## 8. The tests

**79 tests run in 0.10 seconds. That is a great feedback loop.** No real database, no real Claude, no network calls. Engineers will actually run the tests because they are fast.

**The `RuleEvaluatingEvaluator` test is the cleverest one.** It implements the same algorithm we tell Claude to use. So if our prompt drifts away from what Claude actually does, this test catches the difference without spending API credits.

**One test fixture is broken in a sneaky way.** `build_swap_arb_snapshot` says in its comment "8 long-held positions across UTC midnight". But if you read the actual times, the trades open at 18:10 and close at 23:55 on the same UTC date. They never cross midnight. So the rollover rule never fires on this fixture. The test still passes because the threshold is "score >= 75" and the other 3 rules fire (3/4 = 75). If anyone tightens the assertion to "score == 100" they will be confused. Fix the fixture so the times actually span midnight.

**No test for what happens when Claude breaks.** What if Claude raises? What if it returns garbage? What if it returns 3 answers when we asked for 4? None of these are tested.

**No test for thread safety.** We run 4 Claude calls in parallel using one shared HTTP client. The Anthropic library is thread safe today, but if that ever changes silently we would not catch it.

**No test that the prompt stays identical across calls.** The prompt cache only works if the cached part is byte for byte identical between requests. If we accidentally inject a timestamp or an account ID into the system prompt, every "cache hit" becomes a miss and the bill goes up. A simple test that the system prompt is the same for two different snapshots would lock this in.

**No test for the real Anthropic wrapper.** We only test the fake. We should write one test that uses a fake HTTP response and confirms we send the right shape and parse the right fields.

**No test for real MongoDB.** We use mongomock which is close but not identical to real Mongo, especially around dates and time zones. One test against a real Mongo container in CI would help.

**No CI.** No GitHub Actions, no GitLab CI, nothing that runs the tests automatically on every change. Tests have to be remembered.

## 9. Documentation

**context.md is the best file in the project.** Clear, complete, opinionated. Anyone joining can read this and start working.

**The evaluation report (`done-todos/evaluation-report.md`) is too positive.** It says things like "complete and working" and "ready to plug into the real Anthropic API". That hides the real gaps I am listing here. A reader picking this up cold will think the system is more done than it is. Add a "Known limitations" section that lists the holes (no auth, no logs, no error containment, no scheduler).

**The MT5 admin notes are out of date.** They mention a `DealReason` field and a planned change that we never adopted. Either remove the section or mark it "not adopted".

**`context.md` says we have 74 tests; we have 79.** Small but the doc is supposed to be the source of truth so it should be right.

**No CHANGELOG, no CONTRIBUTING, no proper architecture doc.** context.md mostly serves as the architecture doc, which is fine. A CHANGELOG would help once more than one person works on this.

**The message we drafted to Alex lives only in chat history.** That should be in a tracked file (something like `done-todos/messages-to-alex/2026-05-09-data-fields-ask.md`) so future you and future Alex can both find it.

## 10. Production readiness

This is the weakest area. Here is the gap list.

- No logs, anywhere
- No request IDs that you can trace through the system
- No metrics (how slow is Claude today, how much are we spending, how many errors)
- No alerts (if the 06:00 scheduled scan fails, who knows?)
- No auth on the HTTP endpoint
- No rate limiting
- No retry policy tuned for production
- No graceful shutdown
- No Dockerfile, no deployment files
- No secrets management beyond a `.env` file
- No database backup plan
- No runbook for "Claude is down at 12:00 UTC, what do we do?"

The PRD mentions Sentry, Prometheus, and Docker, which are all the right tools, but none are in the build yet. The PRD's tone makes it sound like these are part of MVP. They are not.

## 11. Code style

The code reads well. Comments are short and useful. No dead imports. No copy-paste. Schemas are typed end to end. Tests are well structured. Two tiny things:

- A helper for converting MongoDB documents to JSON sits in `api.py` but it is really about MongoDB shape, so it belongs in `db/repo.py`.
- Two helpers (`is_test_mode`, `has_results`) are unused in production. Decide: delete or document.

**The honest takeaway.** This is solid work for Phase A and you would not be embarrassed to show K. It is also further from production than the evaluation report suggests. Three things to fix first:

1. **Engine error containment.** One Claude failure should not destroy the entire account analysis.
2. **Replace the synchronous Flask + 4 thread design** with a proper queue and worker pool before the scheduler ships in Phase B.
3. **Add at least basic logs and an API key on the endpoint** before this is reachable from outside the local network.
