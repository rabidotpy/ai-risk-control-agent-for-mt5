## 1. The big question: program detection vs. LLM detection

### 1.1 What the question really means

When a detector looks at a trader's data and decides "this looks like latency arbitrage", **who is doing the thinking?** There are three honest options.

| Option                | How it works                                                                                                                                                            | Example                                                                                                                                            |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. Pure program**   | Hard rules and math, written by us. No LLM involved in detection.                                                                                                       | "If more than 70% of trades close in under 60 seconds AND win rate > 85%, flag it."                                                                |
| **B. Pure LLM agent** | We hand the LLM the raw data + a written instruction ("you are a risk officer, here are the trades, decide if this is suspicious"). LLM produces a verdict and a score. | We feed the last 200 deals as JSON and a prompt. LLM replies with `{score: 87, reason: "..."}`.                                                    |
| **C. Hybrid**         | Program does the heavy filtering and number crunching. LLM only writes the explanation and (optionally) helps with edge cases the program is unsure about.              | Program flags the case with score 82. LLM is asked: "explain in 5 bullets" and "is there anything in the comments field that changes the picture?" |

### 1.2 Side-by-side comparison

| Criterion                              | A. Pure Program                               | B. Pure LLM                                                                               | C. Hybrid (recommended)                                                                                      |
| -------------------------------------- | --------------------------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **Auditability**                       | Excellent — every line of logic is in code    | Poor — LLM reasoning is opaque, can change between runs                                   | Good — the score and decision are in code; LLM only adds text                                                |
| **Reproducibility**                    | Same input always gives same output           | Different output possible even with same input (temperature, model updates)               | Score is deterministic; explanation may vary                                                                 |
| **Cost per scan**                      | Near zero                                     | High — every account costs LLM tokens. Thousands of accounts × every 6 hours = real money | Low — LLM only runs on already-flagged cases (small number)                                                  |
| **Speed**                              | Milliseconds per account                      | Seconds per account, plus rate limits                                                     | Fast scan, slow only on the few flagged ones                                                                 |
| **Handling new tricks we never coded** | Bad — only catches what we wrote rules for    | Good — can spot patterns it was trained on                                                | Medium — program catches known, LLM can be asked to "look for anything unusual" on flagged or random samples |
| **Compliance / regulator review**      | Strong — "here is the rule, here is the math" | Weak — "the LLM said so" is not a defense                                                 | Strong — the action-triggering score is rule-based; LLM is only a writer                                     |
| **Risk of false accusations**          | Controlled by tuning thresholds               | High if LLM hallucinates                                                                  | Low — LLM never decides actions                                                                              |
| **Ease of debugging**                  | Easy — read the code, find the bug            | Hard — change a prompt, hope it helps                                                     | Easy for scoring; LLM output is a separate concern                                                           |
| **Tuning by non-developers**           | Medium — change weights in config             | Easy — change the prompt                                                                  | Both available                                                                                               |

### 1.3 Recommendation

**Go with Option C — Hybrid.** Concretely:

- **The score and the action recommendation are produced by a normal program** using the parameters in §2 below. No LLM involved in the number.
- **The LLM agents (already defined in v2 §9) only handle text:** evidence summary, email draft, CS script.
- **One small extra LLM use is allowed:** an "anomaly second opinion" agent that runs on cases scored 60–74 (Medium band only). It receives the structured signals and answers a single yes/no: _"Based on these signals, do you see anything the rules might be missing?"_ Its answer is logged but **does not change the score**. It is used to learn where rules are blind, not to make decisions.

**Why this wins for our project:**

- Regulators and the broker's compliance team will accept rule-based scoring. They will not accept "the LLM said 87".
- It costs very little — we only pay LLM tokens on flagged cases, not on every account every 6 hours.
- We keep the door open to learn from the LLM's "second opinion" log over time. If the second-opinion agent often spots things the rules miss, we add new rules. The LLM becomes a **rule-discovery tool**, not a runtime judge.

**This question must still be confirmed by the team lead** before we lock the design. See §5.

## 2. Concrete Parameters of Risk Creation

This is the master list of every signal that contributes to a risk score. It is organized in two ways:

- **§2.1 by data source** — so Alex knows exactly which fields his APIs must return.
- **§2.2 by case** — so each detector knows which parameters to read and weigh.

A "parameter" here means: a single measurable value, derived from raw broker data, that can move the risk score up.

### 2.2 Parameters grouped by case (which detector uses what)

| Case                        | Primary parameters (heavy weight) | Secondary parameters (light weight) |
| --------------------------- | --------------------------------- | ----------------------------------- |
| **A. Latency Arbitrage**    | 12, 13, 15, 16, 19, 20, 41, 42    | 1, 4, 6, 17, 18, 43, 44             |

| **D. Bonus / Credit Abuse** | 23, 24, 29, 30, 31, 39            | 32, 33, 34, 36, 37, 38              |

Each detector's config file will list these parameter IDs, their normalization range, and their weight. **No magic numbers in code** — everything is in config so risk officers and compliance can tune without redeploying.

### 2.3 How a parameter becomes a sub-score

For every parameter we define three things in config:

```yaml
parameter_12:
  name: "pct_deals_under_60s"
  normalize:
    type: "linear_clamp"
    floor: 0.30 # below this → contribution 0
    ceiling: 0.85 # above this → contribution 1
  weight_per_case:
    latency_arbitrage: 0.25
    scalping_violation: 0.10
```

The detector reads the parameter, normalizes it (0..1), multiplies by the case weight, and sums. The final 0–100 score is the weighted average rescaled. This is the **deterministic formula** referenced in v2 §6.

### 2.4 Parameters are also LLM input (for the explainer agent)

When the Evidence Summarizer Agent (v2 §9.1) writes a summary, the **only data it sees is this parameter list with values**. It never sees raw deals or customer names. This keeps prompts small, cheap, and safe from prompt injection.

## 3. How the parameters feed the chosen engine

Because we recommend the Hybrid engine (§1.3), the data flow becomes very clear:

```
Raw broker data (Alex's APIs)
        │
        ▼
Parameter Extractor  ←  config from §2.3
        │
        ▼
44 numeric parameters per (account, window)
        │
        ├──►  Rule-based Scorer  →  0–100 score  →  Telegram alert
        │                                               │
        │                                               ▼
        └──►  Evidence Summarizer LLM  ─────────►  Officer reads summary
                                                        │
                                                        ▼
                                    [Officer taps action button]
                                                        │
                                                        ▼
                                                 Action Executor
```

**Important:** the parameter extractor is the **single source of truth**. The scorer reads from it. The LLM reads from it. Nothing else reaches the LLM. This is the safety boundary.
