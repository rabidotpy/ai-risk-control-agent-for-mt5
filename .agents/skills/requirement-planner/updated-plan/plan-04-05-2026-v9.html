1. The scope of the LLM (so you know what I'm building) / 大模型的边界(让你了解我在做什么)

English

What the LLM does:

  • Reads a JSON bundle of one trader's recent activity (no personal info).
  • Decides if the data shows one of four risk patterns: Latency Arbitrage, Scalping Violation, Swap Arbitrage, Bonus / Credit Abuse.
  • Writes a short analysis with the specific data points it relied on.
  • Drafts a 6-line Telegram message and a customer email.

What the LLM never does:

  • It never sees personal data (name, email, phone, raw IP).
  • It never executes anything. A human officer taps a button in Telegram, then my service calls your API.

That's the whole agent. Everything below is what I need from your side to make it possible.

中文

大模型做的事:

  • 读取某位交易者近期活动的 JSON 数据包(不含任何个人信息)。
  • 判断数据是否符合四种风险模式之一:延迟套利、剥头皮违规、隔夜利息套利、赠金/信用滥用。
  • 写一段简短分析,标注它依赖的具体数据点。
  • 起草一条 6 行的电报消息和一封发给客户的邮件。

大模型绝不做的事:

  • 不接触任何个人信息(姓名、邮箱、电话、明文 IP)。
  • 不执行任何动作。必须由风控官员在电报里点按钮,我的服务再去调用你的 API。

这就是这个代理的全部。下面列出的内容,都是我为了实现它需要你支持的部分。

2. Data I need to READ from your side / 我需要从你这边读取的数据

English

Per uid, on a schedule or when an event fires, I'll pull a bundle. For each item I explain why I need it.

Data Status Why I need it
account — login_id, uid, group, leverage, balance, equity, credit, currency, kyc_level AVAILABLE Baseline context for every case. Group tells me the min-hold rule for Scalping (R2).
deals — deal_id, symbol, volume, side, open/close time, prices, profit, swap AVAILABLE The core evidence for all four risks.
orders — order_id, request_time, execution_time, request_price, fill_price CONFIRM I need the gap between request and execution to detect Latency Arbitrage (R1). MT5 stores it; I need to know if your API exposes both timestamps.
open_positions — symbol, side, volume, open_time, current_pnl, swap_accrued AVAILABLE Needed to spot offsetting positions across linked accounts (Swap Arbitrage R3).
withdrawals — id, amount, currency, state, submitted_at CONFIRM Bonus/Credit Abuse (R4) hinges on withdrawals right after a bonus. Withdrawals usually live in your CRM/back-office, not MT5.
deposits — id, amount, currency, state, time CONFIRM Same reason as withdrawals — used to compute deposit-to-bonus ratio for R4.
bonuses — id, type, amount, granted_at, terms (required_turnover, expiry_at) NEW I need the bonus terms (required turnover, expiry) to detect churn-trading. I assume nothing exposes this today.
logins — uid, ip_hash, device_fp_hash, country_code, ts NEW Used to detect linked accounts (R3, R4). Critical: I need the IP and device fingerprint hashed before they reach me, never raw.
quote_ticks — symbol, ts (ms), bid, ask CONFIRM Latency Arbitrage (R1) needs millisecond ticks to compare against order timing. Tick storage is heavy — please tell me the resolution and retention you can give me.
news_window — symbol, ts, scheduled events (name, time, severity) NEW Latency arbitrage spikes around news. Helpful to suppress false positives. Lowest priority — can come in v2 if you don't have it.
crm_profile (no PII) — uid, segment, language, country_code, kyc_level, prior_warnings_count, linked_uids_count NEW Gives the LLM context (e.g. "language=zh" so the email is in Chinese). Strictly no PII — only categorical + counts.
One transport question: for the five event triggers (new withdrawal submitted, abnormal short-term profit, high-frequency trading, news event window, suspicious bonus usage) — can you push them to me as webhooks? If not, I'll poll. Webhook is simpler and cheaper; I just need an HMAC secret.

数据 状态 为什么需要

3. Actions I need to CALL into your side / 我需要回调到你这边的动作

English

When the human officer taps a button in Telegram, my service calls one of these. I'll always pass an idempotency key so retries are safe.

Action Status Why I need it
pause_withdrawal(withdrawal_id, reason, case_id, idem_key) CONFIRM Most common action for R4. Must be reversible.
restrict_open_new_position(login_id, reason, case_id, idem_key) AVAILABLE MT5 read-only flag on the trading account. Used for R1, R2.
suspend_trading(login_id, reason, case_id, idem_key) AVAILABLE Disable / lock the account. Used for severe cases. I'll add a 2-step confirm in Telegram before calling.
send_email(uid, subject, body, case_id, idem_key) NEW Critical PII rule: I send only uid + subject + body. Your backend resolves uid → email address. My service must never see the address.
assign_cs_followup(case_id, cs_user, due_at) NEW So a customer-service agent picks up the case in their queue. Could be a row in your CRM.
update_case_state(case_id, state, note) NEW For Watch Only / Ignore / Escalate buttons. State only — no platform side-effect. Could live entirely in my service if you'd prefer.
adjust_profit(deal_id, delta, reason, case_id, idem_key) CONFIRM For profit clawback on confirmed abuse. If this is manual back-office today, I'll just file a request instead of calling — please tell me which.
Two cross-cutting requests:
  1. Idempotency keys on every write. If my service retries because of a network blip, I don't want to pause the same withdrawal twice.
  2. One auth method for both read and write — whatever you prefer (API key, mTLS, short-lived token). Tell me what fits your stack.

动作 状态 为什么需要

两个通用请求:
  1. 所有写入接口都加幂等键。我的服务因为网络抖动重试时,不应该把同一笔出金暂停两次。
