## 1. The problem (in one paragraph)

BWG runs MT5. Some traders abuse the platform in four specific ways. Today, detection is manual and slow. We want an AI assistant that watches every account, recognises the four abuse patterns from real trading data, and pushes a single short Telegram message to a human officer with action buttons. The human approves; we call your APIs to execute. Nothing happens automatically.

---

## 2. The four risks (definitions with one example each)

These are the only risks in MVP scope (PRD §3).

### R1 — Latency Arbitrage

Trader exploits a faster external price feed. Opens a position with us before our price catches up, closes seconds later for near risk-free profit.
**Example:** EURUSD ticks 1.07320 → 1.07338 on a fast feed at 10:21:00.100. Trader buys 1 lot with us at 10:21:00.250 at 1.07321. Closes 14s later at 1.07338 = +$17. Repeats this 200 times in a day with 91% win rate.

### R2 — Scalping Violation

Trader breaks the broker's minimum holding-time rule (per MT5 group, e.g. 60s).
**Example:** Account in group `Pro-Raw-USD` (min hold = 60s). 62 of 80 daily trades close in under 30s, totaling +$1,840.

### R3 — Swap Arbitrage

Trader holds offsetting positions across linked accounts so net market exposure is ~0 but they collect positive overnight swap.
**Example:** Account A long 5 lots AUDJPY, Account B (same KYC, same IP) short 5 lots AUDJPY. Held across the swap-credit cutoff every night for 30 nights. Profit comes entirely from swap.

### R4 — Bonus / Credit Abuse

Trader misuses a bonus: churn-trading to fake turnover, withdrawing right after the bonus, or running a multi-account ring to claim the same bonus repeatedly.
**Example:** Customer deposits $10, claims $500 bonus, runs 200 round-trip trades with cumulative P&L of −$3 over 6 hours, then files a $480 withdrawal.

## 3. The data we need from you (the input to the LLM)

For each scan or event, we collect a per-uid bundle. Naming below is **assumed** — please confirm or rename.

| Data bundle            | What it contains                                                                         | Used by which risk |
| ---------------------- | ---------------------------------------------------------------------------------------- | ------------------ |
| `account`              | login_id, uid, group, leverage, balance, equity, credit, currency, kyc_level             | All                |
| `deals`                | deal_id, symbol, volume, side, open_time, close_time, prices, profit, swap               | R1, R2, R3, R4     |
| `orders`               | order_id, request_time, execution_time, request_price, fill_price                        | R1                 |
| `open_positions`       | symbol, side, volume, open_time, current_pnl, swap_accrued                               | R3                 |
| `withdrawals`          | id, amount, currency, state, submitted_at                                                | R4                 |
| `deposits`             | id, amount, currency, state, time                                                        | R4                 |
| `bonuses`              | id, type, amount, granted_at, terms (required_turnover, expiry_at)                       | R4                 |
| `logins`               | uid, ip_hash, device_fp_hash, country_code, ts                                           | R3, R4             |
| `quote_ticks`          | symbol, ts (ms), bid, ask                                                                | R1                 |
| `news_window`          | symbol, ts, events (name, scheduled_at, severity)                                        | R1                 |
| `crm_profile` (no PII) | uid, segment, language, country_code, kyc_level, prior_warnings_count, linked_uids_count | All                |

**Strict rules on this data:**

- No PII in any field. IPs and device fingerprints arrive **hashed**. Names, emails, phone numbers are never returned.
- Timestamps in ISO-8601 UTC. Tick timestamps in millisecond precision.
- Pagination via `next_cursor` on list endpoints.

## 4. The LLM prompt (one prompt, parameterised by risk)

We send the LLM the per-uid bundle plus a short instruction. Same prompt template for all four risks; only the `target_risk` and parameter focus differ.

```
You are a risk-control analyst for BestWing Global MT5.

You will receive a JSON object with one trader's recent trading
data (no personal information).

Your job:
1. Decide whether the data shows the pattern of {target_risk}.
2. If yes, identify the specific data points (by id) that support it.
3. Write a detailed analysis a human officer can verify.
4. Write one short Telegram message (max 6 lines) summarising the case.
5. Suggest a single primary action.

Rules:
- Use only the data provided. Do not invent numbers.
- Refer to the customer only by `uid`. Never use any name, email, or address.
- Output strictly the JSON schema in the response section. No prose outside JSON.

Target risk: {target_risk}   (one of: latency_arbitrage, scalping_violation,
                              swap_arbitrage, bonus_abuse)
Window: {window}             (e.g. last_24h)
UID: {uid}
Data: { ...the bundle from §3... }
```

## 7. The actions (Telegram tap → callback → your POST API)

### 7.1 The flow

1. Officer taps a button in Telegram.
2. Telegram sends a callback to my service.
3. My service verifies: officer is on the allow-list, case state allows this action, idempotency key is fresh.
4. For critical actions (Suspend, Close Account, Profit Adjustment): we send a "Confirm?" reply and wait for a second tap.
5. My service calls your POST API with an idempotency key.
6. Audit row is written before and after the call.

### 7.2 The POST APIs I will call (assumed names — please confirm)

| Telegram button       | POST API call                                                     | Notes                                                                  |
| --------------------- | ----------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Pause Withdrawal      | `pause_withdrawal(withdrawal_id, reason, case_id, idem_key)`      | Reversible                                                             |
| Restrict Opening      | `restrict_open_new_position(login_id, reason, case_id, idem_key)` | Reversible                                                             |
| Full Suspend          | `suspend_trading(login_id, reason, case_id, idem_key)`            | 2-step confirm                                                         |
| Generate Email + Send | `send_email(uid, subject, body, case_id, idem_key)`               | You resolve uid → email server-side; my service never sees the address |
| Assign CS             | `assign_cs_followup(case_id, cs_user, due_at)`                    | Internal                                                               |
| Watch Only            | `update_case_state(case_id, "watching", note)`                    | Internal                                                               |
| Ignore                | `update_case_state(case_id, "ignored", note)`                     | Internal                                                               |
| Escalate              | `update_case_state(case_id, "escalated", note)`                   | Internal                                                               |
| (Profit clawback)     | `adjust_profit(deal_id, delta, reason, case_id, idem_key)`        | 2-step confirm                                                         |
