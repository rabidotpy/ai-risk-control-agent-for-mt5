## Why this new plan file exists

This v4 file is created because of one important rule the user confirmed on 04-05-2026:

> **We never give a user's personal information to the LLM. When we need to send an email to a user, we call an API with a `uid` and the `email content`. That API looks up the user's email address and sends the message.**

This is a hard privacy and safety boundary. v2 §9.2 (Email Agent) and v3 §2.4 already moved in this direction, but v4 makes it explicit, removes any ambiguity, and updates the data flow, the API list, and the LLM prompts to match. Once this rule is in code, we cannot accidentally leak names, emails, phone numbers, or KYC data into an LLM prompt — even if a developer makes a mistake later.

v4 keeps everything from v1, v2, and v3 by reference. Read v3 first.

## 1. The PII boundary (the rule, in plain words)

**The rule:** No personally identifiable information (PII) ever enters an LLM prompt.

**What counts as PII for this product:**

- Customer first name, last name, full name
- Email address
- Phone number
- Postal address
- Date of birth
- KYC document numbers (passport, ID, tax id)
- IP addresses (treated as PII because they can identify a person)
- Device fingerprints (treated as PII for the same reason)
- Bank account / card numbers
- Free-text notes written by customers or about customers

**What is NOT PII and may go to the LLM:**

- A `uid` (an opaque internal user id; useless to outsiders)
- A `case_id`
- Numeric parameters from v3 §2 (counts, ratios, percentages, durations)
- Symbol names (EURUSD, XAUUSD, etc.)
- Risk score and risk band
- Action names (e.g. "pause_withdrawal")
- Template IDs and template variable names
- Language code (e.g. "en", "zh") — needed so emails are written in the right language

**Why this matters:**

- LLM providers may log prompts. We do not want customer names in those logs.
- A leaked prompt cannot be used to identify a real person if it only contains opaque ids and numbers.
- Compliance and data-protection rules (GDPR-style) become much easier to satisfy.
- Prompt-injection becomes nearly impossible because no customer-controlled free text reaches the model.

## 2. What the LLM is allowed to see — and not see

For each LLM agent from v2 §9, here is the new strict input shape.

### 2.1 Evidence Summarizer Agent

**Allowed input (JSON):**

```json
{
  "case_id": "C-2026-04-30-00128",
  "uid": "u_8a3f0b2",
  "risk_type": "latency_arbitrage",
  "score": 87,
  "band": "high",
  "window": "last_24h",
  "parameters": {
    "pct_deals_under_60s": 0.82,
    "win_rate_sub_minute": 0.91,
    "avg_holding_seconds": 14.3,
    "symbol_concentration": 0.78,
    "news_window_trade_ratio": 0.45
  }
}
```

**Forbidden input:** customer name, email, IP, country, KYC fields, raw deal rows, raw comments.

### 2.2 Email Agent

**Allowed input (JSON):**

```json
{
  "case_id": "C-2026-04-30-00128",
  "uid": "u_8a3f0b2",
  "language": "en",
  "template_id": "withdrawal_under_review",
  "variables": {
    "withdrawal_amount": 980.0,
    "withdrawal_currency": "USD",
    "expected_review_business_days": 3
  }
}
```

**Forbidden input:** customer first name (the template uses a generic salutation like "Dear Customer" or pulls the name from CRM **after** the LLM is done), email address, any identifying detail.

**Output from the LLM:** a JSON object with `subject` and `body` strings — both written using the variables and the template, in the correct language. The body contains no PII because it had none to begin with.

### 2.3 Customer Support Agent

**Allowed input (JSON):**

```json
{
  "case_id": "C-2026-04-30-00128",
  "uid": "u_8a3f0b2",
  "risk_type": "bonus_abuse",
  "action_taken": "withdrawal_paused",
  "language": "en",
  "customer_tier": "standard"
}
```

**Forbidden input:** customer name, contact details, anything else.

**Output:** allowed phrases, forbidden phrases, follow-up checklist. Generic — addressed by `uid`, never by name.

## 3. How email sending now works

### 3.1 The two-step flow

```
Step 1 (LLM):  generate_email_draft(uid, template_id, language, variables)
               → returns { subject, body }   ← no PII inside

Step 2 (API):  send_email(uid, subject, body, case_id)
               → backend looks up the email address for uid
               → backend sends the message
               → returns delivery status
```

**Key point:** the LLM produces text that is **addressed to a `uid`, not to an email address**. Our system never sees the email address. The backend resolves `uid → email_address` at send time, inside the broker's secure backend. This means:

- The email address never touches our service code paths that interact with the LLM.
- The email address never enters logs we control.
- Rotating customer email is invisible to us — backend handles it.

### 3.2 What the officer sees in Telegram

Before sending, the officer sees a preview:

```
Case C-2026-04-30-00128
Recipient: u_8a3f0b2  (resolved at send time)
Language: en
Template: withdrawal_under_review

Subject: Update on your recent withdrawal request
Body:
Dear Customer,

We are currently completing routine checks on your account.
Your withdrawal request of USD 980.00 is on hold while
this review is in progress. We expect to update you within
3 business days...

[Approve & Send]   [Edit]   [Cancel]
```

The officer never sees the raw email address either, unless they explicitly open the CRM in another tool. Inside our system, identity is always `uid`.

### 3.3 Edit flow

If the officer taps **Edit**, they get a text box. The edited text **never goes back to the LLM**. It goes straight to the `send_email` API. This stops a feedback loop where edits could leak PII back into the model.

## 4. Changes to the data flow diagram

The diagram from v3 §3 is updated. The new node is the **PII Filter**, which sits between the parameter extractor and any LLM agent.

```
Raw broker data (Alex's APIs)
        │
        ▼
Parameter Extractor
        │
        ▼
44 numeric parameters per (account, window)
        │
        ├──► Rule-based Scorer ──► 0–100 score ──► Telegram alert
        │                                              │
        │                                              ▼
        ▼                                     [Officer taps a button]
   PII Filter (drops names, emails, IPs)              │
        │                                              ▼
        ▼                                     LLM call (summary / email / CS)
   LLM-safe payload (uid + numbers only) ◄────────────┘
        │
        ▼
   LLM output (text only, no PII added because none was given)
        │
        ▼
   Officer approves
        │
        ▼
   send_email(uid, subject, body, case_id)
        │
        ▼
   Backend resolves uid → email address → sends → returns status
        │
        ▼
   Audit log records: case_id, uid, template_id, prompt_hash, output_hash, delivery_status
```

The audit log stores **hashes** of the prompt and output, not the raw text by default. Raw text is only kept when explicitly enabled for compliance review, behind extra access control.

## 5. Changes to the API list

In v2 §4.2, write API #18 was `send_email(client_id, template_id, vars, case_id)`. v4 changes it.

### 5.1 Updated API #18

**Old:** `send_email(client_id, template_id, vars, case_id)` — backend renders template.

**New (v4):**

```
send_email(uid, subject, body, case_id, idempotency_key)
```

- `uid` — opaque user id.
- `subject` — already-rendered subject string (LLM output, officer-approved).
- `body` — already-rendered body string (LLM output, officer-approved).
- `case_id` — for audit linking.
- `idempotency_key` — prevents double-send on retry.

The backend (Alex's service) is responsible for: looking up the email address for `uid`, applying any company-required header/footer, sending via SMTP/relay, and returning `{message_id, status}`.

### 5.2 New supporting API request

We also need a small read API so the **officer's preview** can show a non-PII recipient label (e.g. "u_8a3f0b2 — VIP segment" or "u_8a3f0b2 — language: en"). This **does not return** the email address.

**New API #21:**

```
get_uid_label(uid) → { uid, segment, language, country_code }
```

- Returns only minimal, non-identifying metadata for display.
- Country code is allowed because it is too coarse to identify a person on its own. If compliance disagrees, we drop it.

### 5.3 What is removed

- We no longer need any API that returns the customer's email address to our service. If such an API exists, our service must not call it.

## 6. Changes to the LLM agents

Concrete edits to v2 §9 (no need to repeat the whole section here).

### 6.1 Evidence Summarizer (v2 §9.1)

- Input is now strictly the JSON in §2.1.
- The phrase "5 example deals (anonymized)" from v2 is **removed**. Even anonymized rows can carry timing patterns that re-identify a person when combined with public market data. Only aggregated parameters go in.

### 6.2 Email Agent (v2 §9.2)

- Input is now strictly the JSON in §2.2.
- Templates are written so they never need a first name. Default salutation: `"Dear Customer"` (or its translation). If a template variant **with** a name is ever required, the backend (not the LLM) substitutes it after the LLM is done, server-side.
- Output goes to the new `send_email(uid, subject, body, case_id, idempotency_key)` API.

### 6.3 Customer Support Agent (v2 §9.3)

- Input is now strictly the JSON in §2.3.
- The CS playbook references the customer as "the customer for case C-2026-04-30-00128" — never by name.

### 6.4 Common safety upgrade (extends v2 §9.4)

Add three rules to the existing eight:

- **Rule 9 — PII Filter is mandatory.** Every LLM call goes through a single function `prepare_llm_payload(case)` that white-lists fields. No other path to the LLM is allowed.
- **Rule 10 — Static check.** A unit test runs on every PR: it scans known LLM call sites and fails the build if a forbidden field name (e.g. `email`, `first_name`, `ip`) appears in the payload schema.
- **Rule 11 — Prompt logging hashes by default.** Raw prompt and output text are not logged unless a compliance flag is set on the case. Hashes are always logged so we can prove what was sent.
