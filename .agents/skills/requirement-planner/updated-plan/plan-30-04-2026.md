## 1. Project In a Nutshell (50 words)

An internal, AI-assisted risk-control system for an **MT5 broker** (fiat/commodities, not crypto). It scans accounts on schedule and on event triggers, detects abusive trading patterns (latency arbitrage, scalping, swap arbitrage, bonus abuse), scores risk 0–100, alerts a human officer via a **Telegram bot**, and executes approved actions (pause withdrawal, restrict trading, email customer) with full audit logs.

## 4. Risk Cases — Definitions, Detection Steps, Decisions

The four MVP cases below are derived directly from PRD §3. For each case I list:

- **What it is** (plain English)
- **Detection signals & curl-equivalent steps**
- **Score contributors** (feeding the 0–100 scorer)
- **Decisions & actions** the officer can take

### 4.1 Case A — Latency Arbitrage

**What:** Trader exploits a slower broker price feed vs. a faster reference feed (e.g., they see EURUSD move on a fast feed before our feed reflects it; opens position pre-move, closes seconds later for near-risk-free profit).

**Detection steps (curl flow):**

1. `list_deals(login_id, last_24h)` → filter deals where `close_time - open_time < 60s` and `profit > 0`.
2. For each candidate, `get_quote_history(symbol, open_time-2s, close_time+2s)` from our feed.
3. Cross-check vs. external/reference feed (if available; otherwise flag pattern only).
4. `list_orders(login_id, ...)` → look for execution latency (slippage_ms) consistently in trader's favor.
5. `get_account_pnl(login_id, 7d)` → high win-rate (>85%) on sub-minute trades.

**Score contributors:** sub-minute trade ratio, avg holding time, win rate on those trades, profit-per-trade vs. spread, repeated symbol concentration.

**Decisions/actions:** Watch Only → Restrict Opening → Suspend Trading → Profit Adjustment (clawback) → Email + CS follow-up.

### 4.2 Case B — Scalping Violation

**What:** Account violates broker's scalping policy (e.g., minimum holding time per group). Often defined at MT5 group level.

**Detection steps:**

1. `get_account(login_id)` → read `group` and its scalping policy (e.g., min hold = 60s).
2. `list_deals(login_id, last_7d)` → count deals violating the min-hold threshold.
3. Compute violation ratio and aggregate violating profit.
4. `get_crm_profile(client_id)` → check prior warnings.

**Score contributors:** % of deals under threshold, total violating profit, repeat offender flag.

**Decisions/actions:** Warning email → Restrict Opening → Profit Adjustment → Suspend.

### 4.3 Case C — Swap Arbitrage

**What:** Trader holds offsetting positions across correlated symbols / accounts to harvest positive swap (rollover interest) with negligible market risk; commonly held across the swap-credit timestamp.

**Detection steps:**

1. `list_open_positions(login_id)` and across `client_id`'s linked accounts.
2. Identify hedged exposure: net delta ≈ 0 across correlated pairs (e.g., AUDJPY long + similar-beta short).
3. `list_deals(login_id, 30d)` → measure swap income vs. trading P&L.
4. Detect repeated open just before swap-credit cutoff and close just after.
5. `get_crm_profile(client_id)` → look for multiple linked accounts (same KYC, same IP via `list_logins`).

**Score contributors:** swap income / total profit ratio, hedge persistence around swap cutoff, multi-account linkage.

**Decisions/actions:** Restrict Opening on hedged pair → Adjust swap credits → Suspend → Email.

### 4.4 Case D — Bonus / Credit Abuse

**What:** Client uses a bonus/credit (deposit-match, no-deposit) in ways that violate terms — e.g., minimum-volume churn, withdrawal-before-turnover, sibling/duplicate accounts to multiply bonuses.

**Detection steps:**

1. `list_bonuses(client_id)` → active bonus & terms.
2. `list_deals(login_id, since=bonus_granted_at)` → measure required turnover vs. actual; check trades that look like volume-churn (open/close fast, near-zero P&L).
3. `list_withdrawals(client_id, state=pending)` → withdrawal soon after bonus grant.
4. `list_deposits(client_id)` → small deposit followed by large bonus claim.
5. `list_logins(client_id)` + cross-client IP/device match → duplicate-account ring.

**Score contributors:** turnover gap, churn-trade ratio, withdraw-after-bonus speed, duplicate-account count.

**Decisions/actions:** Pause Withdrawal → Reverse Bonus → Restrict Opening → Close Account (for ring fraud) → Email + CS.

## 5. Risk Scoring (0–100)

PRD bands: 0–39 Low, 40–59 Watch, 60–74 Medium, 75–89 High, 90–100 Critical.

**Approach for MVP:** weighted-sum of normalized signals per case type. Each detector emits sub-scores (0–1); a case-level weighted sum is rescaled to 0–100. LLM is **not** used for scoring in MVP — only for evidence summary and email drafting (deterministic scoring is auditable; LLM scoring is not).

**Auto-routing of levels:**

- Low → log only.
- Watch → silent watchlist, no Telegram push.
- Medium → Telegram alert, recommend "Watch Only" / "Pause Withdrawal".
- High → Telegram alert, recommend "Restrict Opening" + "Pause Withdrawal".
- Critical → Telegram alert, recommend "Full Suspend" + escalation tag.

All recommendations still require **human approval** before write APIs are called.

## 6. Case Lifecycle & State Machine

```
detected
  → pushed_to_telegram
    → under_review (officer opened)
      → action_executed (one or more write APIs invoked)
        → email_sent (optional)
          → cs_follow_up (optional, with due date)
            → closed
```

Side branches: `ignored`, `escalated`, `false_positive` (must record reason). Every transition writes an audit row: `case_id, from_state, to_state, actor, ts, reason, payload_hash`.

## 7. MVP Implementation Plan (phased)

### Phase 0 — Foundations (no Alex APIs needed)

- Repo bootstrap: Python + FastAPI, Postgres, Redis, APScheduler, aiogram, Docker.
- Define internal data models matching §3.1.
- Build a **mock data adapter** so detectors can run on synthetic data while waiting for Alex.
- Define the case schema and audit-log table.

### Phase 1 — Detector framework

- Pluggable detector interface: `detect(window, account_batch) -> list[CaseCandidate]`.
- Implement detectors A–D against mock data.
- Implement deterministic scorer with per-case weights (config-driven, not hard-coded).

### Phase 2 — Telegram bot

- Alert push with the 8 PRD fields (Case ID, Account, score, level, type, evidence, suggested action).
- Inline action buttons: View Evidence, Watch Only, Pause Withdrawal, Restrict Opening, Full Suspend, Generate Email, Assign CS, Ignore, Escalate.
- Each button calls an internal handler; write actions are queued and require a confirm tap.

### Phase 3 — Action executor + audit

- Wrap each write API (13–20) behind a single `execute_action(case_id, action, params)` that: (a) checks state, (b) calls Alex's API, (c) records audit, (d) updates case state.
- Idempotency keys per `(case_id, action)`.

### Phase 4 — LLM agents

- **Email Agent:** templated drafts ("under review" tone, no accusations). LLM fills variables; officer must approve before send.
- **CS Agent:** generates a script bundle (allowed phrases, forbidden phrases, follow-up checklist).
- Evidence summarizer: turns raw data into a short Telegram-friendly bullet list.

### Phase 5 — Scheduling & event triggers

- APScheduler cron at 00/06/12/18.
- Event-trigger webhook receiver for: withdrawal submit, news-event window, etc. (assumption: Alex will publish webhooks or we poll).

### Phase 6 — Hardening

- Rate limits on write APIs, circuit breaker on Alex's backend, Sentry, Prometheus metrics, role-based bot auth (only allow-listed Telegram user IDs).

## 9. Risks & Mitigations

| Risk                                         | Impact                      | Mitigation                                                                            |
| -------------------------------------------- | --------------------------- | ------------------------------------------------------------------------------------- |
| API spec changes after Alex delivers         | Rework of adapter           | Isolate all Alex calls behind a single `BrokerClient` interface                       |
| False positives causing bad customer impact  | Reputational, financial     | Mandatory human approval on all write actions; "watch only" default for new detectors |
| LLM hallucinated evidence in emails          | Compliance breach           | Templated emails only; LLM fills slots, officer signs off                             |
| Telegram bot account compromise              | Catastrophic (mass actions) | Allow-list of TG user IDs + 2-step confirm + per-action rate limits                   |
| Latency-arb detection without reference feed | Detector ineffective        | Phase 1 ships pattern-only heuristics; Phase 2 adds reference feed when available     |
| Scope creep into AIO behaviors               | Wrong product target        | Keep product-name TBD field abstract; confirm with Alex before product-specific code  |
