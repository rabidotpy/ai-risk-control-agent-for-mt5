# Plan — 2026-06-01: post-rule-refactor snapshot + profitable_client_pattern

## Why this plan exists

Previous plans (April–May 2026) described a system where Claude was evaluating
the rules in prompts, and where many sub-rules were blocked on data Alex had
not yet delivered. Both situations changed materially:

1. Rule evaluation was moved from Claude prompts to deterministic Python
   (commit 023a4cf, 2026-05-17). Same input always produces the same score.
2. The data Alex was already sending turned out to support most of the
   "blocked" rules once we paired In/Out deals correctly. As of today, 19
   of 21 sub-rules are LIVE.
3. A 5th risk type was added on 2026-06-01: `profitable_client_pattern`. It is
   an operational signal for the dealing desk (route to A-book), not a
   compliance signal. Inspired by analysis of real client account 250031.
4. The earlier research files (`data_requirements_matrix.md`,
   `Values_to_sub_rules_mapping.md`) described the BLOCKED state from before
   the refactor and are now rewritten in this same commit.

This plan file captures the current state as of 2026-06-01 so future readers
do not have to reconstruct it from commit history.

## Architecture (as it actually exists)

1. `POST /analyse_risk` accepts a batch envelope `{"snapshots": [...]}` per
   the FastAPI route. The envelope is forgiving about variations (singular
   `snapshot`, single-object value, single-element list) but strict on
   unknown top-level keys.
2. Per snapshot, per risk type:
   a. Run prescreen (`app/services/prescreen.py`). If the rule cannot
      plausibly fire on this snapshot, persist a zero-score audit row and
      skip the LLM.
   b. Run the deterministic Python evaluator (`app/rules/<risk>.py`).
      Produce the score from the count of TRUE sub-rules.
   c. If the score crosses `llm_narrate_min_score` (default 60), call
      Claude for narrative output: `summary`, `behavior_summary`,
      `evidence_description_list` (4 plain-English items aimed at customer
      support and the broker's risk officer).
   d. Persist the finding in `RiskEvaluation`. Audit trail is always
      complete, regardless of any gate.
3. After all snapshots are processed, the outbound filter
   (`filter_high_risk_accounts`) drops accounts whose max risk score is
   below `callback_min_score` (default 60). Result goes to Alex's callback
   URL.

## Default thresholds

| Setting | Default | Purpose |
| --- | --- | --- |
| `prescreen_enabled` | True | Skip LLM for risks that cannot plausibly fire |
| `llm_narrate_min_score` | 60 | Below this, use templated narrative (no LLM call) |
| `callback_min_score` | 60 | Below this, drop the account from the outbound callback |

## What needs to be cleaned up next (technical debt)

1. **Schema cleanup**: remove `bid_at_open` and `ask_at_open` from the
   `Trade` schema. No live rule uses them; they are still required and Alex
   defaults them to zero.
2. **Provisional thresholds**: scalping R1 (`>= 25`) and bonus R2 (`>= 8`)
   are rescaled from PRD-literal 24h figures to fit the 6h scan window.
   When Phase B internal aggregation lands, revisit and set them to the
   real 24h values.
3. **A-book action enum**: the `suggested_action` for
   `profitable_client_pattern` reuses compliance strings
   ("restrict_opening_pause_withdrawal") because there is no separate
   action mapping yet. The narrate prompt tells the LLM to frame the
   message as "route to A-book". A clean fix is to add a per-risk action
   mapping later.

## What is NOT in MVP scope

- News-arbitrage detection (PRD §6.3 rule 6) — would need an economic
  calendar feed.
- Real-time per-trade evaluation — current cadence is the scheduled scan
  every 6 hours. Real-time on every trade is a different architectural
  shape and not promised.
- Stop-out / margin-stress detection — would need per-account margin
  level from Alex.
- KYC-driven flags (country, tier, IB chain) — schema does not currently
  accept these.
