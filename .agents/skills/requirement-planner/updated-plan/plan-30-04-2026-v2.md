## 1. What this product is, in simple words

**Product name:** BestWing Global MT5 AI Risk Control Agent MVP.

**What it does, in one paragraph:**
A trading broker (BestWing Global) lets people trade money on the MT5 platform — for example, buy and sell EURUSD, gold, or oil. Some traders try to cheat. They use tricks to win money in ways that are not fair to the broker. Our product watches all the trading accounts, finds the cheaters, gives each suspicious case a score from 0 to 100, and sends an alert to a human officer through a **Telegram bot**. The officer looks at the case and decides what to do: stop the trader's withdrawal, block new trades, send an email, or close the account. The system writes down everything it does so the company has proof later.

**Important facts to remember:**

- This is for **MT5** (a desktop trading platform). It is about real money in normal currencies and commodities. **It is not about crypto.**
- This product is **not AIO**. AIO is a different product. We are building something separate. (The exact place inside the company's product line is still TBD with Alex.)
- The first user is the **Risk Score Officer** (a human). The system never takes a final action by itself. The human always says yes or no first.
- A backend developer named **Alex** will build the APIs we need. Today we do not have those APIs. We have to make educated guesses (assumptions) and check them with Alex later.

This is the part v1 was missing. Before reading the details, look at the flow once:

```
                     ┌──────────────────────────────┐
                     │  MT5 / CRM / Payments / etc. │  ← Alex's APIs
                     └──────────────┬───────────────┘
                                    │ (we call them)
                                    ▼
   ┌───────────┐   ┌──────────────────────────────────────┐
   │ Scheduler │──►│  Detectors (4 rules: latency, scalp, │
   │ (cron     │   │  swap, bonus). Each makes a sub-score│
   │  + events)│   └──────────────┬───────────────────────┘
   └───────────┘                  │
                                  ▼
                     ┌──────────────────────────┐
                     │  Scorer (0–100, 5 bands) │
                     └──────────────┬───────────┘
                                    │
                                    ▼
                     ┌──────────────────────────┐
                     │  Case created in DB      │
                     └──────────────┬───────────┘
                                    │
                                    ▼
                     ┌──────────────────────────┐
                     │  Telegram Bot alert      │ ← Officer sees it
                     └──────────────┬───────────┘
                                    │ (officer taps button)
                                    ▼
                     ┌──────────────────────────┐
                     │  LLM Agents prepare:     │
                     │  • evidence summary      │
                     │  • email draft           │
                     │  • CS script             │
                     └──────────────┬───────────┘
                                    │ (officer approves)
                                    ▼
                     ┌──────────────────────────┐
                     │  Action Executor calls   │
                     │  Alex's write APIs       │
                     └──────────────┬───────────┘
                                    │
                                    ▼
                     ┌──────────────────────────┐
                     │  Audit log writes every  │
                     │  step to the database    │
                     └──────────────────────────┘
```

**Read it like a story:**

1. Every 6 hours, or right after an event, the **Scheduler** wakes up.
2. It asks the **Detectors** to look at trading data from Alex's APIs.
3. Each detector that finds something suspicious gives a **sub-score**.
4. The **Scorer** combines sub-scores into one number from 0 to 100.
5. If the number is high enough, a **Case** is born and saved in our database.
6. The **Telegram Bot** sends a message to the officer.
7. The officer taps "View Evidence" → an **LLM Agent** prepares a short, easy-to-read summary.
8. The officer taps an action button (for example "Pause Withdrawal").
9. The **Action Executor** calls Alex's write API and stops the withdrawal.
10. The **Audit Log** writes down what happened, who did it, and when.
11. If needed, the **Email Agent** writes a draft email and the **CS Agent** prepares a script. The officer must still approve them.
12. The case moves through states until it is **closed**.

## 5. The four risk cases, explained simply

For each case, we answer four questions in plain language:

- **What is the trick?**
- **How do we catch it?** (Which APIs from §4.2 we call, in order.)
- **What signals raise the score?**
- **What can the officer do?**

### 5.1 Latency Arbitrage

- **What is the trick?** The trader uses a faster external price feed than the one our broker uses. They see the price move on the fast feed first, then quickly buy or sell on us before our price catches up. They close the trade a few seconds later with almost guaranteed profit. It is a kind of "I already know the future" attack.
- **How do we catch it?**
  1. Call API #2 (`list_deals`) for the last 24 hours.
  2. Keep only deals that are very short (closed in under 60 seconds) AND profitable.
  3. For each kept deal, call API #10 (`get_quote_history`) to replay our prices around the trade.
  4. (Optional) Compare with a reference feed if we have one.
  5. Call API #3 (`list_orders`) to check if the request-to-execution delay was always in the trader's favor.
  6. Call API #5 (`get_account_pnl`) to check if the win rate over the last 7 days is suspiciously high (above 85%).
- **Signals that raise the score:** ratio of sub-minute trades, average holding time, win rate, profit per trade compared to the spread, the same symbol used over and over.
- **Officer actions (in order of severity):** Watch Only → Restrict Opening → Suspend Trading → Profit Adjustment (take back unfair profit) → send Email → assign CS follow-up.

### 5.2 Scalping Violation

- **What is the trick?** The broker has a rule like "you must hold a trade for at least 60 seconds in this account group". The trader breaks the rule on purpose to make many tiny fast trades.
- **How do we catch it?**
  1. Call API #1 (`get_account`) to read the trader's group and the minimum holding time of that group.
  2. Call API #2 (`list_deals`) for the last 7 days.
  3. Count how many deals were closed faster than the minimum holding time.
  4. Add up the profit from those bad deals.
  5. Call API #12 (`get_crm_profile`) to see if this trader was warned before.
- **Signals that raise the score:** percentage of bad deals, total profit from bad deals, repeat offender flag.
- **Officer actions:** Warning Email → Restrict Opening → Profit Adjustment → Suspend.

### 5.3 Swap Arbitrage

- **What is the trick?** Every day at a certain time, brokers pay or charge "swap" (rollover interest). The trader opens two opposite trades (one buy, one sell on related symbols) so they take almost no market risk, but they collect positive swap. Sometimes they spread these trades across two or more accounts owned by the same person.
- **How do we catch it?**
  1. Call API #4 (`list_open_positions`) for the trader, and for any accounts that look like the same person.
  2. Check if the buy volume and the sell volume cancel each other out (net exposure ≈ zero) on related pairs (for example AUD/JPY long and another pair short with similar behavior).
  3. Call API #2 (`list_deals`) for the last 30 days. Compare swap income vs. real trading P&L. If almost all profit comes from swap, it is suspicious.
  4. Look for trades opened just before the swap-credit time and closed just after.
  5. Call API #12 (`get_crm_profile`) and API #9 (`list_logins`) to find linked accounts (same KYC name, same IP, same device).
- **Signals that raise the score:** swap income divided by total profit, how often the trader holds the hedge across the swap time, number of linked accounts.
- **Officer actions:** Restrict Opening on the hedged pair → Adjust swap credits → Suspend → send Email.

### 5.4 Bonus / Credit Abuse

- **What is the trick?** The broker gives free money (bonus or credit) with rules — for example "you must trade $100,000 in volume before you can withdraw". The trader either fakes the volume with churn trades, withdraws too early, or opens many fake accounts to claim the bonus many times.
- **How do we catch it?**
  1. Call API #8 (`list_bonuses`) to see active bonuses and their rules.
  2. Call API #2 (`list_deals`) since the bonus was granted. Check if the required turnover is real or fake (churn trades make the volume look big but earn nothing).
  3. Call API #6 (`list_withdrawals`) — is there a pending withdrawal soon after the bonus?
  4. Call API #7 (`list_deposits`) — small deposit followed by a big bonus claim?
  5. Call API #9 (`list_logins`) and cross-check IPs and devices across customers — many accounts from the same place is a red flag (a "ring").
- **Signals that raise the score:** turnover gap, churn-trade ratio, speed from bonus to withdrawal, number of duplicate accounts.
- **Officer actions:** Pause Withdrawal → Reverse Bonus → Restrict Opening → Close Account (for ring fraud) → send Email + CS follow-up.

## 6. How the risk score works

We use a **simple, predictable math formula**, not an LLM, to make the score. Why? Because regulators and the officer need to understand and trust the score. An LLM score cannot easily be explained in court. A formula can.

**How a sub-score is made:**
Each detector produces signals. Each signal is normalized to a number between 0 and 1 (where 1 = very bad). Example for latency arbitrage:

- sub-minute trade ratio: 0.7 means 70% of trades are sub-minute → already bad.
- win rate on those trades: 0.92.
- profit-per-trade vs. spread: 0.8.
- repeated symbol concentration: 0.6.

**How sub-scores combine into the final 0–100:**
We do a weighted average. The weights live in a config file, **not** hard-coded. This way we can tune them without changing the code.

`final_score = 100 × (w1·s1 + w2·s2 + w3·s3 + w4·s4) / (w1+w2+w3+w4)`

**Score bands and what happens for each:**

| Score  | Level    | What the system does                                                    |
| ------ | -------- | ----------------------------------------------------------------------- |
| 0–39   | Low      | Quietly logs the case. No alert.                                        |
| 40–59  | Watch    | Adds the account to a watchlist. No alert.                              |
| 60–74  | Medium   | Sends a Telegram alert. Suggests "Watch Only" or "Pause Withdrawal".    |
| 75–89  | High     | Telegram alert. Suggests "Restrict Opening" + "Pause Withdrawal".       |
| 90–100 | Critical | Telegram alert with red flag. Suggests "Full Suspend" + escalation tag. |

**Important:** even at "Critical", **nothing happens automatically**. The officer must tap a button.

## 7. The case lifecycle (the journey of one case)

A case is like a ticket. It moves through states. Every move is recorded.

```
detected
  → pushed_to_telegram
    → under_review        (officer opened it)
      → action_executed   (one or more write APIs were called)
        → email_sent      (optional)
          → cs_follow_up  (optional, with a due date)
            → closed
```

**Side branches:**

- `ignored` — officer decided it was nothing. Reason must be written.
- `escalated` — sent to a senior officer or compliance.
- `false_positive` — the detector was wrong. Reason must be written. This feedback later helps tune the weights.

**Audit row written on every move:**
`case_id, from_state, to_state, actor (officer telegram id), timestamp, reason, payload_hash`.

The `payload_hash` is a fingerprint of the data that was used. If anyone asks "what data did you base this decision on?", we can prove it.

## 8. Step-by-step build plan (Phase 0 to Phase 6)

We build in phases so we can show working software early. Phases 0 and 1 do **not** need Alex; we use mock data. This means we and Alex can work in parallel.

### Phase 0 — Foundations (no Alex APIs needed)

- Set up the repository with the recommended tech stack.
- Create the database tables: `accounts`, `cases`, `case_events`, `audit_log`, `actions_queue`.
- Build a **mock data adapter** that pretends to be Alex's backend. This lets us test detectors with synthetic data right away.
- Wrap all calls to "Alex" behind one Python class called `BrokerClient`. When Alex's real APIs arrive, we change only this class.

### Phase 1 — Detector framework

- Define a small interface every detector follows: `detect(window, accounts) → list of CaseCandidates`.
- Implement the four detectors against mock data.
- Build the deterministic scorer with config-driven weights.

### Phase 2 — Telegram bot

- Send alerts that include all 8 fields the PRD asks for (Case ID, Account ID, Risk Score, Risk Level, Risk Type, Evidence Summary, Suggested Action, plus a "View Evidence" link).
- Add inline buttons: View Evidence, Watch Only, Pause Withdrawal, Restrict Opening, Full Suspend, Generate Email, Assign CS, Ignore, Escalate.
- Every action button shows a "Confirm?" tap before doing anything.
- Allow-list of Telegram user IDs — strangers cannot use the bot.

### Phase 3 — Action executor + audit

- One function `execute_action(case_id, action, params)` does:
  1. Check the current case state allows this action.
  2. Call Alex's write API.
  3. Record the audit row.
  4. Move the case to the next state.
- Use idempotency keys per `(case_id, action)` so a double-tap does not pause the same withdrawal twice.

### Phase 4 — LLM agents

**See §9 for the full deep-dive. This phase is the one v1 explained badly.**

### Phase 5 — Scheduling and event triggers

- APScheduler runs the four scans at 00, 06, 12, 18.
- An HTTP endpoint receives webhooks from Alex's platform when a withdrawal is submitted, when a news event hits, etc. If Alex cannot send webhooks, we fall back to polling every minute.

### Phase 6 — Hardening

- Rate limits on every write API.
- Circuit breaker: if Alex's backend is down, we stop sending writes and alert the team.
- Sentry for errors, Prometheus for metrics.
- Two-step confirm on critical actions (Suspend, Close Account).
- Backups of the audit log.

## 9. Phase 4 explained in detail (LLM agents)

**Why we need LLMs at all:**
The detectors give us **raw numbers and rows from a database**. A human officer reading this on a phone in Telegram would be overwhelmed. The LLM's job is to **translate raw data into short human language**, **draft emails the officer can approve**, and **prepare scripts for CS people**. The LLM **never decides anything**. It only writes text.

**Three separate agents, each with one job:**

### 9.1 Evidence Summarizer Agent

- **What it does:** Turns the raw signals from a detector into a 5-line summary the officer can read in 10 seconds.
- **Input it gets:**
  ```
  case_id, risk_type, score, top_signals (e.g. "82% of trades sub-minute",
  "win rate 91%", "average hold time 14 seconds"), 5 example deals (anonymized).
  ```
- **Output it gives:**
  ```
  • This account closed 218 trades in 24h.
  • 82% were closed under 60 seconds.
  • Win rate on those trades is 91% — far above normal (~52%).
  • Most trades are on EURUSD around news minutes.
  • Looks like classic latency arbitrage.
  ```
- **How the human controls it:** The summary appears in the Telegram alert. If the officer wants the full data, they tap "View Evidence" which opens a webview with the raw rows. The LLM cannot hide anything; the raw data is always one tap away.
- **Safety:** The LLM is given **only** structured fields, never free text from customers. This avoids prompt-injection from a customer support note, etc.
- **Where it sits in the flow:** Between the Scorer (§3 step 4) and the Telegram Bot (§3 step 6). It runs **once per case**, when the case is created.

### 9.2 Email Agent

- **What it does:** Drafts a customer-facing email after the officer asks for one. The wording must be polite and **never accuse the customer**. Examples of acceptable tone: "Your account is currently under review", "We have temporarily paused your withdrawal while we complete checks".
- **Input it gets:**
  ```
  case_id, customer_first_name, language (from CRM profile),
  template_id (e.g. "withdrawal_under_review"),
  variables (e.g. withdrawal_amount, expected_review_days).
  ```
- **Output it gives:** A draft email body and subject. The draft is **not sent**. It appears in Telegram as a preview with three buttons: "Approve & Send", "Edit", "Cancel".
- **Why we use templates:** A pure free-text LLM email is dangerous (it can hallucinate facts or accuse the customer). So the LLM is allowed to **fill in slots** in pre-approved templates. It can rephrase a sentence for tone, but it cannot add new facts.
- **How the human controls it:** Nothing leaves the system without "Approve & Send". The send action goes through the Action Executor (Phase 3), so it is logged.
- **Where it sits in the flow:** It runs only when the officer taps "Generate Email" in Telegram (§3 step 7).

### 9.3 Customer Support (CS) Agent

- **What it does:** When the officer assigns the case to a CS person, this agent generates a small "playbook" for that CS person. The playbook tells them what they can say, what they cannot say, and which boxes to tick during follow-up.
- **Input it gets:**
  ```
  case_id, risk_type, action_taken (e.g. "withdrawal paused"),
  customer language, customer tier.
  ```
- **Output it gives:**
  ```
  Allowed phrases:
    "Your account is under routine review."
    "We will update you within 3 business days."
  Forbidden phrases:
    "Fraud", "abuse", "we caught you", any legal threat.
  Follow-up checklist:
    [ ] Customer contacted within 24h
    [ ] Customer documents requested if needed
    [ ] Status updated in case
  ```
- **How the human controls it:** The CS person receives the playbook and uses it. If they want to deviate, they must escalate back to the officer.
- **Where it sits in the flow:** It runs when the officer taps "Assign CS" in Telegram. The case enters the `cs_follow_up` state (§7).

### 9.4 Common rules for all three LLM agents

These rules apply to every LLM call we make. They are the "guardrails":

1. **No decisions.** LLMs only produce text. They never call write APIs.
2. **Structured input only.** We pass JSON-shaped fields, not raw chat history. This blocks prompt injection.
3. **Templated outputs.** Email and CS scripts are slot-filled into pre-approved templates.
4. **Always reviewable.** Every LLM output is shown to a human before any action happens.
5. **Logged.** The full prompt, the model name, the version, and the output are saved in the audit log next to the case. If the LLM output ever causes a problem, we can replay it.
6. **Model choice is config-driven.** We can switch from OpenAI to Azure OpenAI to a local model without changing code.
7. **Token budget.** Each agent has a max token limit so we cannot run away with cost.
8. **Fallback.** If the LLM API is down, the system still works: the officer sees the raw signals (no nice summary) and can still take actions.

## 10. Risks and how we will handle them

| Risk                                                        | Impact                      | Mitigation                                                                                     |
| ----------------------------------------------------------- | --------------------------- | ---------------------------------------------------------------------------------------------- |
| Alex's API spec changes after we build                      | Rework needed               | Hide all calls behind `BrokerClient`. Only that one file changes.                              |
| False positives upset real customers                        | Financial + reputation      | Human approval is required on every write. New detectors start in "Watch Only".                |
| LLM writes a wrong fact in an email                         | Compliance breach           | Templates with slot fill only. Officer must approve. Full prompt is logged.                    |
| Telegram bot account is hacked                              | Catastrophic — mass actions | Allow-list of Telegram user IDs. Two-step confirm on critical actions. Per-action rate limits. |
| Latency arbitrage cannot be proven without a reference feed | Detector is weak            | Phase 1 ships pattern-only signals. Phase 2 adds reference feed if available.                  |
| Scope creep into AIO behaviors                              | Wrong product target        | Keep all product-specific code behind a flag. Confirm with Alex before adding any.             |
| Prompt injection through customer fields (e.g. CRM notes)   | Hijacked LLM output         | LLMs receive only structured fields, never free text from customer-controlled sources.         |
