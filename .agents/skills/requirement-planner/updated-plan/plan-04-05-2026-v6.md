## Phase A — Problem framing

### Task 1: Define the problem with clarity, in detail

- I need to write down what is broken today at the broker without this system.
- I need to write what "risk" specifically means in the MT5 context — not crypto, not generic fraud, but the four PRD activities.
- I need to write the cost of doing nothing: how much money the broker bleeds per month from latency-arb alone, how slow human-only review is, what regulators expect.
- Output: one clear paragraph I can paste at the top of any deck.

### Task 2: Get clarity around why solving this matters

- I will write only what is necessary. No filler.
- I must make sure each point is unique and does not repeat task 1.
- Points to cover:
  - Revenue protection (direct money lost to abuse).
  - Regulatory exposure (broker license risk if abuse goes unchecked).
  - Customer trust (clean traders pay for the abuse via wider spreads).
  - Scalability (a human alone cannot review thousands of accounts every 6 hours).
- Output: a short "why now" section, 5 bullet points max.

### Task 3: Get clarity on how we can solve this

- I will state the high-level approach: **Hybrid**. Rule engine produces the score. LLM produces the text. Human approves every action.
- I will explain why I am not choosing pure rules (too rigid, misses new tricks) and not choosing pure LLM (not auditable, expensive, regulators will not accept).
- I will list the minimum building blocks I need end-to-end:
  - Parameter Extractor
  - Rule-based Scorer
  - LLM agents (Evidence Summarizer, Email Agent, CS Agent)
  - Telegram Bot console
  - Action Executor
  - Audit log
- Output: a one-page architecture sketch.

### Task 6: Decide the polling cadence per data source

- I need to choose, per data source: long poll, short poll, or on-demand pull only.
- Working defaults I will propose:
  - Deals / orders / positions → on-demand during scan (no polling).
  - Withdrawals (pending) → short poll every 60s if no webhook.
  - News calendar → short poll every 60s (small payload).
  - Quote history → on-demand only when a deal is being investigated.
  - CRM profile → cache aggressively, refresh on case open.
- I will document my reasoning so Alex can push back if any cadence is wrong for his side.

### Task 7: Decide how often the system itself runs scans

- Scheduled scans per PRD: every 6 hours at 00:00, 06:00, 12:00, 18:00.
- Window strategy per scan: 6h, 24h, 7d, 30d.
- Per-uid event-driven scan on demand whenever an event arrives.
- I need to confirm with Alex that the broker can sustain the read-API load four times per day across all accounts.

### Task 8: Define the WebSocket / push events from BWG

- I will list the five event types from PRD §2:
  - Withdrawal request submitted
  - Abnormal short-term profit detected
  - High-frequency trading detected
  - News-event trading window triggered
  - Suspicious bonus usage detected
- I will define the assumed event payload shape (event_type, uid, ref_id, timestamp, payload object).
- I will define security: HMAC signature on every push, verify it on receipt.
- I will define idempotency: dedupe by `event_type + ref_id`.
- I will define the fallback: if WebSocket / webhook is not available on Alex's side, I poll instead. The rest of the system does not change.

## Phase D — LLM responsibility

### Task 9: Define what data the LLM analyses and how

- The LLM analyses **only the parameter dictionary** computed by the rule engine — never raw deals, never PII.
- Its job: pick the 3–5 parameters that most strongly support the case and write a short bullet summary in plain language.
- I will write the prompt template myself and lock it in config:
  - JSON-only output
  - Max 5 bullets, max 15 words each
  - No invented numbers, no recommended actions, no PII
  - Final verdict line in broker vocabulary (e.g. "Pattern is consistent with latency arbitrage")
- I will also decide whether to ship the optional "second-opinion" LLM for Medium-band cases. My current lean: yes, but log-only — it never changes the score.

## Phase G — Email-send flow

### Task 13: Define the email-send flow (uid + content; backend resolves the address)

- Step 1 — LLM generates draft:
  - Input: `uid`, `template_id`, `language`, `variables` (no PII).
  - Output: `{ subject, body }`.
- Step 2 — Officer reviews preview in Telegram:
  - Buttons: Approve & Send / Edit / Cancel.
  - If Edit: text goes straight to send, never back to the LLM.
- Step 3 — My service calls `send_email(uid, subject, body, case_id, idempotency_key)`.
  - Alex's backend resolves `uid → email address` server-side.
  - Backend sends via SMTP / its email transport.
  - Backend returns `{ message_id, status }`.
- My service never sees the email address.
- Audit row stores `prompt_hash`, `output_hash`, `delivery_status` — raw text only behind a compliance flag.
